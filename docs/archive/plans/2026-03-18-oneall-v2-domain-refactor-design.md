# OneAll v2 域重构与瘦身设计

## 背景与目标

现有 OneAll 平台按照「智能运维门户」思路设计，集成了探针拨测、外部监控集成（Zabbix/Prometheus）、资产中心、统计分析、工具库、知识库、代码管理等多种能力。随着使用场景收敛，现阶段的真实需求更偏向于：

- 以自研探针 + 拨测能力为核心；
- 保留少量可扩展的能力中心（资产、工具）；
- 用一个独立、清晰的告警域承接探针/监控产生的事件；
- 砍掉重依赖、重集成、弱价值的模块（外部监控展示、统计大盘、知识库、代码管理等）。

本设计目标是定义 OneAll v2 的领域边界与模块结构，把当前「大而全」的平台收缩为「探针拨测 + 可扩展能力中心 + 告警域」的系统，并为后续实现计划提供清晰的架构基线。

## 范围与取舍

### 保留的核心能力

- 自研探针系统（Go 探针 + Django 后端 `probes` 域）；
- 一次性拨测 / 证书检测（HTTP/TCP/Telnet/WSS/证书等协议）；
- 持续拨测监控任务（监控任务创建、调度执行、结果存储）；
- 资产中心：保留为「可扩展资产能力中心」，一期只实现本地域名资产；
- 工具库：保留为「可扩展工具能力中心」，聚焦通用运维工具执行；
- 基础系统能力：用户认证、角色/权限、基础设置等。

### 明确删除 / 迁出的能力

- Zabbix / Prometheus 监控展示与深度集成：
  - 删除后端 monitoring 领域下对外部监控的集成代码；
  - 删除前端 integrations 相关页面和 Dashboard/Monitoring 中的外部监控视图。
- 统计分析大盘：
  - 删除独立 analytics 模块（基于 TimescaleDB 的报表与趋势分析）；
  - 仅在监控任务/结果视图中保留轻量统计查询。
- 知识库：
  - 整体从本项目中移除（后续可迁移为独立系统或服务）。
- 代码管理：
  - 统一从本项目中移除，不再把脚本仓库/在线代码编辑作为 OneAll 能力的一部分。
- 系统设置中的重集成项：
  - 移除/废弃 Zabbix/Prometheus/ITSM/LDAP 等强耦合配置项；
  - 收敛为平台基础配置 + 告警通道基础参数 + 探针基础配置。

### 新增 / 强化的方向

- 新增独立告警域（alerts app）：统一管理告警通道、规则和事件，解耦监控/探针与通知；
- 资产中心 & 工具库重构为「可扩展能力中心」：
  - 抽象资产源/工具类型的扩展协议；
  - v2 中仅实现最小内置实现（本地域名资产 + 若干工具），为未来插件预留结构；
- 监控任务和探针执行统一走「事件 → 告警」链路，而非直接在监控内部发送通知。

## 后端领域划分（Django Apps）

### 保留并重构的应用

1. `core`
   - 职责：认证、用户、角色、审计等基础域。
   - 策略：尽量不做大调整，仅在权限模型上对新域（alerts、资产/工具扩展）增加必要权限。

2. `probes`
   - 职责：探针节点管理、任务下发、执行状态管理。
   - 目标：
     - 统一「一次性拨测」和「持续拨测任务」的执行通路；
     - 保持与 gRPC Gateway/探针协议的兼容，避免重写通信层。

3. `monitoring`
   - 当前职责：监控任务管理、执行结果存储、部分外部监控集成。
   - v2 目标职责：
     - 仅承担内部监控任务域：任务定义、调度策略、执行结果、任务状态机；
     - 删除外部监控集成相关代码：
       - 移除 `monitoring.integrations` 目录及相关 models/services/tasks；
       - 清理对应的 API、序列化器、路由。
     - 将告警触发逻辑从 monitoring 中抽离，改为发出 domain event 给 alerts。

4. `assets`
   - 当前职责：资产中心（多源资产同步与管理）。
   - v2 目标职责：
     - 聚焦「资产核心域」+「可扩展资产源」：
       - 内置模型：本地域名资产（如 `DomainAsset`，包含域名、系统、负责人、互联网类型、告警联系人等）；
       - 扩展协议：定义 `AssetSource` 抽象接口，用于描述 Zabbix/IPMP/工单主机等外部资产源；
       - 同步任务：通过 `AssetSyncTask` 记录从外部源同步的任务和结果。
     - v2 实现：
       - 只实现 `DomainAsset` 等本地模型；
       - 保留 AssetSource/AssetSyncTask 的数据结构与接口，但不实现具体外部源同步逻辑。

5. `tools`
   - 当前职责：工具库（脚本工具管理与执行），可能夹带部分代码管理能力。
   - v2 目标职责：
     - 纯粹的「工具定义 + 执行记录」：
       - `Tool`：工具定义（名称、描述、类型、参数 schema、执行方式等）；
       - `ToolExecution`：执行记录（输入参数、输出摘要、状态、耗时、执行节点）。
     - 工具类型扩展：
       - 抽象工具类型（本地执行、远程调用、探针执行等），但 v2 只实现已有的精简工具；
     - 移除代码管理相关能力：不再提供在线脚本仓库、版本控制等功能。

6. `settings`
   - 当前职责：系统设置，包含用户、权限、系统日志、告警通道配置、外部系统接口配置等。
   - v2 目标职责：
     - 只保留平台基础设置：
       - 平台基础信息（名称、Logo、主题等）；
       - 极少量与全局行为强相关的配置（如默认时区、默认探针心跳周期基线）；
     - 告警相关设置迁移到新建的 alerts 域（如通道配置、通知模板等）；
     - 外部系统接口配置（ITSM、Zabbix、Prometheus 等）整体移除或标记为废弃。

### 新增应用

1. `alerts`
   - 职责：统一的告警域管理。
   - 模型：
     - `AlertChannel`：单一告警渠道配置（邮件、企业微信、钉钉、飞书、自定义 Webhook 等）；
     - `AlertRule`：告警规则，可以绑定到监控任务、资产、探针节点或某个标签；
     - `AlertEvent`：一次告警实例（触发时间、关联任务/资产、规则、发送结果等）。
   - 服务接口：
     - 面向 monitoring/probes 提供统一入口，例如：
       - `alerts.services.evaluate_and_raise(check_result)`：
         - 根据监控结果和相关上下文（任务、资产、节点）匹配规则；
         - 创建 AlertEvent 并调度异步发送任务。
   - 任务：
     - 使用 Celery 异步发送告警：
       - 读取未发送或需要重试的 AlertEvent；
       - 根据绑定的 AlertChannel 发送通知，记录结果。

2. （可选）精简观察性/报表域
   - 可新建轻量级 `observability` app 或重用 `analytics` 名称但彻底重写：
     - 不依赖 TimescaleDB；
     - 仅提供：
       - 按任务查询最近 N 次监控结果；
       - 计算简单的成功率与平均响应时间；
     - 也可以选择直接将这些查询逻辑放入 monitoring/probes 的 service 层，而不新建 app。

### 删除 / 迁出的应用

1. `analytics`
   - 动作：
     - 从 INSTALLED_APPS 和 URL 路由中移除；
     - 删除对应代码目录、任务、服务；
     - 数据表处理：
       - 如有保留价值，先执行导出脚本（CSV/快照库）；
       - 确认后通过迁移删除表。

2. `knowledge`
   - 动作：
     - 完整移除知识库域的 models, api, services；
     - 清理管理后台/菜单/权限中与知识库相关的部分；
     - 同样先评估是否需要导出数据。

3. `monitoring.integrations`
   - 动作：
     - 删除目录 `backend/src/apps/monitoring/integrations` 及所有引用；
     - 删除或迁移其中的 models（如仅用于外部监控集成的实体）；
     - 清理相关 API、任务、服务，以及前端对接点。

## 核心数据流重设计

### 一次性拨测（Detection）

1. 前端：
   - 在 `frontend/src/pages/detection` 中提供一次性拨测界面，支持 HTTP/HTTPS/TCP/Telnet/WSS/证书等协议；
   - 用户选择探针节点/节点策略、填写目标和参数，发起一次性拨测请求。

2. 后端：
   - 接口层接受一次性拨测请求，构造一个临时检测任务实体（不必持久化为长期监控任务）；
   - 调用 probes 域以统一方式选择探针节点（随机/指定/按标签策略）；
   - 通过 gRPC Gateway 将任务下发到探针节点执行；
   - 探针执行完成后，将结果通过 gRPC 回传给后端：
     - 记录为一次 `CheckResult`（或统一的 `MonitoringResult`）；
     - 触发内部 domain event `CheckCompleted`，传递给 alerts 域进行告警评估（一次性拨测默认可以不触发告警，或仅在用户显式要求时触发）。

### 持续拨测监控（Monitoring）

1. 任务生命周期：
   - 用户在监控模块创建长期监控任务（包括目标、协议参数、频率、阈值、关联资产/系统等）；
   - Celery Beat 定时根据任务配置生成具体执行实例，交给 probes 域执行；
   - 探针执行完成后回传结果，写入 `MonitoringResult` 表。

2. 状态与事件：
   - monitoring 域负责维护任务状态（启用/停用、最近一次状态、连续失败次数等）；
   - 每次执行完成后，构造 `CheckCompleted` 事件（包含任务 ID、资产 ID、探针 ID、结果状态、指标数据等）；
   - 将事件提交给 alerts 域的评估接口，由 alerts 决定是否产生 AlertEvent 并发通知。

### 告警（Alerts）

1. 事件评估：
   - alerts 域通过服务函数接受 `CheckCompleted` 事件；
   - 根据 AlertRule 匹配：
     - 按监控任务匹配（如某任务连续失败 >= N 次）；
     - 按资产匹配（某资产下所有任务的失败率过高）；
     - 按探针节点匹配（节点本身状态异常）；
   - 满足规则时创建 AlertEvent 并触发异步发送任务。

2. 通知发送：
   - Celery worker 拉取待发送的 AlertEvent；
   - 根据绑定的 AlertChannel 类型调用相应发送逻辑（SMTP、Webhook 等）；
   - 更新 AlertEvent 的发送状态和结果信息。

3. 与监控/探针的解耦：
   - monitoring/probes 不直接持有任何「发送邮件/微信」的依赖；
   - 所有通知相关配置集中在 alerts 域，后续接入公司统一告警中心只需要在 alerts 层增加新的渠道实现。

## 资产中心与工具库的扩展设计

### 资产中心（Assets）

1. 内置模型（v2 范围）：
   - `DomainAsset`：
     - 字段示例：域名、所属系统、互联网类型（内/外网）、负责人、告警联系人、标签等；
     - 用作监控任务和拨测目标的核心对象。

2. 扩展协议：
   - `AssetSource` 抽象：
     - 标识某个外部资产源（Zabbix/IPMP/工单主机等）；
     - 存储访问配置与同步策略（但 v2 中不提供具体实现）。
   - `AssetSyncTask`：
     - 记录从某个 AssetSource 同步资产的任务；
     - 保存执行时间、结果摘要、错误信息等。

3. v2 实现策略：
   - 保证数据库与代码结构允许未来添加 Zabbix/IPMP 等实现；
   - 但本版本默认不创建任何外部源配置，也不在 UI 中暴露相关配置表单；
   - 前端资产中心聚焦于域名资产管理，额外为「外部资产源」预留占位入口（如显示为空/未配置状态）。

### 工具库（Tools）

1. 内置模型：
   - `Tool`：
     - 字段示例：名称、描述、类型（本地脚本、API 调用、探针执行等）、参数 schema、可见范围等；
   - `ToolExecution`：
     - 字段示例：关联 Tool、输入参数、执行节点（本地/探针）、执行开始/结束时间、状态、输出摘要/日志。

2. 扩展机制：
   - 抽象不同类型工具的执行策略（例如本地 Python 执行、HTTP 调用、在探针上执行命令等）；
   - v2 只支持现有的、风险可控的工具类型；
   - 后续如需增加新工具类型，只需实现对应执行策略并在前端增加适配配置 UI。

3. 移除代码管理：
   - 所有「通用脚本仓库/在线代码编辑/版本控制」相关功能不再由 OneAll 承担；
   - 若有迁移需求，通过一次性导出脚本/配置的方式交给独立系统处理。

## 前端结构与导航调整

### 路由与页面变更

基于当前 `frontend/src/pages` 目录，目标变更如下：

- 删除路由/页面：
  - `analytics/*`：统计分析大盘；
  - `knowledge/*`：知识库相关页面；
  - `integrations/*`：外部监控展示页面。

- 保留并重构：
  - `detection`：
    - 聚焦一次性拨测和证书检测，提供统一、多协议的检测界面；
    - 使用统一的后端接口和探针任务流。
  - `monitoring`：
    - 管理持续拨测任务（创建、编辑、启用/停用）；
    - 结果详情页展示最近 N 次结果、简单成功率/平均响应时间等轻量统计。
  - `probes`：
    - 探针节点列表、状态、运行时长等；
    - 对接后端 probes 域的节点管理接口。
  - `assets`：
    - 聚焦本地域名资产的 CRUD；
    - 对外部资产源仅保留占位/信息展示，不提供配置表单。
  - `tools`：
    - 工具列表与执行记录；
    - 不再包含脚本仓库/代码管理 UI。
  - `dashboard`：
    - 由「统一监控大盘」收缩为轻量总览：
      - 探针在线情况；
      - 监控任务数量与成功率概览；
      - 未处理告警数量等。
  - `settings`：
    - 收敛为用户/角色管理、基础平台信息配置；
    - 告警通道/规则配置迁移到新的告警模块页面。

- 新增/调整：
  - `alerts` 页面：
    - 告警事件列表与详情；
    - 告警规则和通道配置（也可以部分嵌入 settings）。

### UX 原则

- 面向 v2 的定位，把导航和文案统一收紧为「探针拨测平台」；
- 所有被砍掉的功能不再出现在主导航中，避免给用户错误预期；
- 对于未来扩展点（外部资产源、额外工具类型），通过简单的占位和提示说明「当前版本未启用」。

## 迁移与风险控制

由于选择了高强度领域重构，需要在实现层面注意以下几点：

1. 数据迁移
   - 为 `analytics`、`knowledge` 等即将删除的域编写数据导出脚本（如管理命令或 one-off 脚本），将可能有价值的数据导出为 CSV/归档库；
   - 新建 `alerts`/重构 `assets`/`tools` 时，通过 Django migration 为新模型建表，并在必要时写数据迁移脚本平滑迁移旧字段。

2. API 兼容性
   - 列出所有被删除/改签名的 API，评估是否有外部系统依赖；
   - 对于仍有调用方的 API，可以保留一段兼容期，返回明确的错误信息（如 410 Gone + 说明），而不是静默 404。

3. 探针协议稳定性
   - 重构重点集中在应用层领域划分和数据流，不改变探针协议和 gRPC Gateway 行为；
   - 通过端到端回归（探针接入 → 拨测 → 结果上报 → 告警）验证兼容性。

4. 渐进式落地
   - 可以先实现新的 `alerts` 域与重构后的 `monitoring` 流程，再由前端切换到新接口；
   - 等新流程稳定后，再逐步删掉 `analytics`、`knowledge`、`monitoring.integrations` 等老模块，最后清理数据库表。

## 小结

本设计将 OneAll v2 的核心定位为「探针拨测 + 可扩展能力中心 + 独立告警域」，通过：

- 删除外部监控展示、统计大盘、知识库、代码管理等非核心能力；
- 重构 `monitoring`、`assets`、`tools` 等领域边界；
- 新增 `alerts` 告警应用并重塑数据流（探针/监控 → 事件 → 告警）；
- 精简前端路由和导航；

达成「功能瘦身 + 领域结构清晰化」的目标，为后续按需增加插件化能力留下空间。
