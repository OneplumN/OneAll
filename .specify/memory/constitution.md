# One Pro Constitution

## Core Principles

### 原则一：独立价值交付
- **要求**: 每个用户故事必须单独记录于 `spec.md` 并能独立通过验收，计划与任务需确保故事之间互不阻塞。
- **要求**: 在 `tasks.md` 中，所有任务必须标注所属用户故事与并行属性，确保可以按优先级逐一交付。
- **理由**: 保持增量发布能力，让团队随时交付可验证的用户价值。

### 原则二：计划先行与透明假设
- **要求**: 在执行任何实现前，必须使用 `/speckit.plan` 生成计划，并补齐技术上下文与未决问题；未解项需标记为风险并跟踪至关闭。
- **要求**: `plan.md`、`spec.md`、`tasks.md` 的修改需同步互相引用，并在摘要中记录最新决策与假设。
- **理由**: 通过前置设计暴露依赖与风险，降低返工并确保所有人共享同一认知。

### 原则三：验证驱动交付
- **要求**: 每项改动必须在实施前确认可执行的验证方式（自动化测试、脚本或可复现步骤），并在对应任务中列出验证责任人。
- **要求**: 新增或修改的功能必须伴随失败到通过的验证记录，保存在 `research.md`、测试文件或提交说明中。
- **理由**: 持续可验证的交付流程才能保障行为一致性与回归效率。

### 原则四：可观测与知识沉淀
- **要求**: 新增行为需提供结构化日志或度量点，记录位置与格式应在计划的技术上下文中声明。
- **要求**: 行为或配置变更完成后必须更新 `quickstart.md`、代理指南或其他运行文档，保持团队知识可追溯。
- **理由**: 可观测性和知识沉淀让问题可诊断、经验可复用。

### 原则五：简洁结构与版本纪律
- **要求**: 项目结构应遵循最小可行边界，新增目录或服务必须在计划的“Structure Decision”与“Complexity Tracking”中说明原因。
- **要求**: 任何破坏性接口或行为调整必须提供迁移步骤、更新版本号，并在提交信息中链接受影响的文档。
- **要求**: 功能交付完成后必须创建对应的 Git 版本记录（如标签或发布说明），并记录可追溯的数据库备份点以保障回滚能力。
- **理由**: 控制复杂度并保持版本轨迹清晰，保障迭代可回滚。

## 工程约束

- **分支规范**: 功能分支命名遵循 `[###-feature-name]`，所有 `.specify/scripts/bash` 自动化需在对应功能分支运行。
- **规格目录**: 每个功能位于 `specs/[###-feature-name]/`，包含 `plan.md`、`research.md`、`data-model.md`、`quickstart.md`、`contracts/` 与 `tasks.md`。
- **文档同步**: 影响运行方式或操作步骤的改动必须在同一变更中更新相关文档，并在 `agent` 指南或 `quickstart.md` 中记录。
- **日志格式**: 运行时日志应使用结构化格式（JSON 或键值对），覆盖关键路径的成功与失败事件。
- **Git 版本更新**: 每个用户故事或功能合并前需完成语义化版本评估，创建对应 Git 标签或发布记录，并在提交说明中指明受影响的文档与迁移步骤。
- **数据库备份**: 涉及持久化数据的变更需在部署前完成一次可恢复的数据库备份，记录备份时间、位置与恢复验证结果。

## 开发流程

- **阶段 0（研究）**: 通过 `/speckit.plan` 完成 Summary 与 Technical Context，列出未决问题并标记责任人。
- **阶段 1（设计）**: 完成 `data-model.md`、`contracts/` 与 `quickstart.md`，并在 Constitution Check 中验证五项原则均无违例。
- **阶段 2（计划执行）**: 使用 `/speckit.tasks` 生成任务，确保每个用户故事具备独立的实现与验证路径。
- **阶段 3（交付与验收）**: 完成实现后更新文档、执行验证并记录结果，完成 Git 版本标记与数据库备份，并在评审中确认记录可追溯。

## Governance

- **优先级**: 本宪章优先于其他流程或工具文档；出现冲突时以宪章规定为准。
- **修订流程**: 修订需在独立变更中说明影响范围、迁移计划与建议版本号，获得全体维护者评审通过后方可生效。
- **版本策略**: 宪章采用语义化版本；新增原则或重大治理变动视为 MINOR 或 MAJOR 级别，文字澄清记为 PATCH。
- **合规审查**: 每个功能在计划提交与代码合并前必须对照 Constitution Check；维护者需每季度进行一次宪章回顾以确认原则仍适用。
